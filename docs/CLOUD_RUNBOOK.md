## 1. One-time setup (skip if already done)

### A. SSH key (laptop)

```bash
ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N "" -C "yourname@team_builder"
cat ~/.ssh/id_ed25519.pub   # copy the output for next step
```

The empty passphrase (`-N ""`) is intentional — lets scripts SSH non-interactively.

### B. RunPod account

1. Sign up at https://runpod.io (Google sign-in works).
2. Add **$25-50** to your account (Billing → Add Funds).
3. Optional but recommended: complete identity verification → unlocks **Secure Cloud** ($1.50-1.69/hr A100) which is more reliable than Community Cloud (interruptible, ~$0.80-1.20/hr).
4. **Settings → SSH Public Keys** → paste `~/.ssh/id_ed25519.pub` content.

### C. Cloudflare R2 (storage)

1. Sign up at https://dash.cloudflare.com (Google sign-in works).
2. Left nav → **R2 Object Storage** → **Subscribe to R2** (10 GB free + $0.015/GB after; **zero egress fees ever**, which beats AWS S3).
3. Add a payment method when prompted.
4. **Create bucket** → name `team-builder-data` → location: **Automatic**.
5. **Manage R2 API Tokens** → **Create API Token**:
   - Name: `team_builder_sync`
   - Permissions: **Object Read & Write**
   - Bucket scope: `team-builder-data` only
   - TTL: forever (or 90 days)
   - **Copy these IMMEDIATELY** — R2 only shows them once:
     - Access Key ID
     - Secret Access Key
     - Endpoint (`https://<account-id>.r2.cloudflarestorage.com`)

6. Note your **Cloudflare Account ID** (R2 dashboard top-right). Format:
   32-char hex string. Used in the endpoint URL.

Make the repo public (https://github.com/Adith-Rai/team_builder/settings → Change visibility), OR generate a fine-grained Personal Access Token with read-only `Repository contents` for that repo. Public is simpler since the code has no secrets.

## 2. Per-run: deploy a fresh pod

### Deploy

1. RunPod console → **Deploy** (or **Pods → + Deploy**).
2. Filter for **A100 80GB** (PCIe or SXM). **A100 SXM 80GB at $1.50/hr** is the volume sweet spot — slightly faster than PCIe 40GB at $1.69/hr.
3. **Pod template**: select **PyTorch 2.x** (RunPod's official template — has CUDA 12.4, Python 3.11, jupyter, ssh pre-installed).
4. **Container disk: 150 GB** (default 5 is way too small; we need 104 GB memmap + repo + ckpts).
5. **Volume disk: 0 GB** (we use R2, not network volume).
6. **TCP port 22** in "Expose TCP ports" (SSH).
7. **HTTP port 8888** in "Expose HTTP ports" (Jupyter, optional).
8. SSH terminal access: **ON**. Encrypt volume / Start Jupyter: **OFF**.
9. Click **Deploy**, wait 30-60s for status to go to "Running".
10. Click **Connect** → copy the SSH command (looks like `ssh root@<ip> -p <port> -i ~/.ssh/id_ed25519`).

### First-time fix: install SSH key on pod

If you get "Permission denied (publickey)" when you SSH in, the pod doesn't yet have your public key in `/root/.ssh/authorized_keys`. The fix:

1. In RunPod dashboard, click **Connect** → **Web Terminal** (browser-based).
2. Paste this (with your actual public key from `~/.ssh/id_ed25519.pub`):

```bash
mkdir -p /root/.ssh && \
echo "ssh-ed25519 AAAA...your-actual-public-key... user@host" >> /root/.ssh/authorized_keys && \
chmod 600 /root/.ssh/authorized_keys && \
echo "key installed"
```

3. Now SSH from your laptop works. (Some templates / regions auto-inject the key; many don't.)

⚠ **Quirk**: `test_policy.py` runs on CPU and is **extremely slow on RunPod's multi-tenant CPU host** (5+ min instead of 30s local). The throughput on training (which uses GPU) is unaffected. You can kill the CPU test with `kill <pid>` — it's not blocking.

## 4. Per-run: launch real BC training

```bash
cd /workspace/team_builder/pokemon-ai-starter/pokemon-ai/src

nohup python -u train_bc.py --use-transformer --compile \
  --memmap-dir data/datasets/human_v8_100k \
  --epochs 50 --batch-size 48 --lr 1e-4 --fp16 \
  --workers 8 --eval-games 0 --val-ratio 0.1 \
  --sched constant --warmup-steps 200 \
  --early-stop-patience 2 \
  --save-every 1000 --run-name v10_cloud_gen9 \
  --device cuda \
  > /workspace/v10_cloud_gen9.log 2>&1 &

echo $! > /workspace/v10.pid
disown
```

### Why these flags (validated Session 48)

| Flag | Why |
|------|-----|
| `--use-transformer` | New arch (Session 47). Without this, falls back to legacy MLP. |
| `--compile` | torch.compile on spatial + temporal stacks. ~10-25% speedup, ~30s warmup at first batch. |
| `--batch-size 48` | Metamon's published default. 80 GB A100 has plenty of headroom but B>48 has diminishing returns. |
| `--lr 1e-4` | Metamon's published default. Constant (not cosine). Don't scale by batch — BC is not LR-sensitive. |
| `--workers 8` | Metamon's default. RunPod A100 pod has 16+ vCPUs; this leaves headroom. |
| `--sched constant` | Matches Metamon's effective behavior (their cosine has eta_min=lr, so it's constant in practice). |
| `--early-stop-patience 2` | Stops after 2 epochs with no val_loss improvement. Metamon convention. |
| `--save-every 1000` | Mid-step ckpts every ~5-10 min on A100. Crash insurance. Each ~240 MB. |
| `--epochs 50` | Just an upper bound; early-stop fires way before. |
| `--fp16` | AMP — mostly for memory headroom. A100 supports tf32 + bf16 too. |
| `--eval-games 0` | Disables in-script BattleAgent eval. Eval done separately via `eval_metamon_competitive.py`. |
| `--val-ratio 0.1` | 10% held-out for val_loss curve. |

### Monitor

```bash
tail -F /workspace/v10_cloud_gen9.log
```

Watch for:
- `[20] loss=... acc=... turns=... elapsed=...` — every 20 batches
- `Epoch N (local 0): train_loss=... train_acc=... val_loss=... val_acc=... time=...` — end of each epoch
- `New best val_loss=...` — when best.pt updates
- `[checkpoint saved at batch ...]` — every 1000 batches
- `[early-stop] patience exhausted; stopping at epoch N.` — clean exit

To detach without stopping training: `Ctrl+C` to kill `tail`, training keeps going. Re-attach with `tail -F /workspace/v10_cloud_gen9.log`.

## 6. Resuming a crashed/stopped run

The training process saves full state (model + optimizer + scheduler) every `--save-every` batches. Resume from any saved ckpt:

```bash
nohup python -u train_bc.py --use-transformer --compile \
  --resume /workspace/team_builder/.../bc/v10_cloud_gen9/mid_step<N>_epoch<E>_batch<B>.pt \
  --memmap-dir data/datasets/human_v8_100k \
  --epochs 50 --batch-size 48 --lr 1e-4 --fp16 \
  --workers 8 --eval-games 0 --val-ratio 0.1 \
  --sched constant --warmup-steps 200 --early-stop-patience 2 \
  --save-every 1000 --run-name v10_cloud_gen9 \
  --device cuda \
  > /workspace/v10_cloud_gen9.log 2>&1 &
```

⚠ **Critical**: do **NOT** add `--lr-restart` if you want to preserve AdamW momentum + cosine schedule progress. With `--lr-restart`, optimizer + scheduler reset (model weights only loaded). Without it, training continues seamlessly.

⚠ **Mid-epoch resume re-iterates the epoch from batch 0**. The DataLoader doesn't track within-epoch position. The model weights and optimizer state continue from the saved point, but the dataset gets reshuffled. Effective extra pass through data — usually positive for BC.

## 9. Quirks + gotchas (validated Session 48)

These are real things we hit in this session. Future sessions will hit them again.

### Setup-time quirks

1. **PyTorch 2.x container vs our requirements.txt**: RunPod's PyTorch template comes with `torch==2.4.0+cu124` pre-installed. Our `requirements.txt` pins `torch==2.2.1`. Forcing the pin would downgrade and break CUDA wheel compat. `cloud_setup.sh` handles this: it filters `torch*` lines out of requirements before running `pip install`.

2. **numpy version conflict**: `poke-env==0.10.0` requires `numpy>=2.0.2`, but our requirements pinned `numpy==1.26.4`. `cloud_setup.sh` filters numpy too (along with torch); pip resolves to numpy 2.4.x which works fine with our code (no NumPy 1.x specific APIs).

3. **Vocab + lookup data files are gitignored**: `data/vocab/*.json` and `data/lookup/move_flags_v1.pt` (~1 MB total) live under `pokemon-ai-starter/pokemon-ai/src/data/` which is broadly gitignored. **Must scp them to the pod** before tests pass. See section 3d.

4. **SSH key not auto-installed on every pod**: Some RunPod templates pick up the account-level SSH key; many don't. If you get "Permission denied (publickey)", use the web terminal to add the key to `/root/.ssh/authorized_keys` (section 2 fix).

5. **R2 token scope rejects ListBuckets**: `aws s3 ls --profile r2 ...` (no bucket) returns `AccessDenied`. R2 tokens are scoped per-bucket. Use `aws s3 ls s3://team-builder-data/ ...` instead.

6. **R2 sync runs faster than residential ETA**: User's connection completed 104 GB upload in ~75 min (not the 24 hr I estimated for typical 10 Mbps). Cloudflare's network is generous on upload.

7. **CPU on RunPod is multi-tenant + slow**: tests that run on CPU (`test_policy.py` uses `torch.device("cpu")`) take 10× longer than local. Skip them or kill them; throughput on GPU training is unaffected.

8. **arch detection bug for Sessions ≤48**: `is_transformer_checkpoint()` originally matched both legacy and new arch because they share `spatial.*`, `temporal.*`, `summary_to_temporal.*`, `value_head.*` prefixes. **Discriminate on `tokenizer.`, `switch_encoder.`, `action_head.`** — these are unique to the new arch (legacy uses `move_net`, `pokemon_net`, `action_encoder`). Fixed in commit after Session 48 eval discoveries.

### Run-time quirks

9. **`cloud_smoke.sh` has a hardcoded 10-minute timeout**: it reports FAIL even when training is fine but slow. Look at the actual loss trajectory in the log, not the script's exit code. Real failures show NaN, Tracebacks, or zero progress.

10. **torch.compile recompiles on first epoch boundary**: you'll see `[1/8] To log all recompilation reasons, use TORCH_LOGS="recompiles"` warnings at epoch transition. Adds ~10-20s overhead, then continues normally. Not a problem.

11. **`pad_sequence` warnings spam the log**: `Support for mismatched src_key_padding_mask and mask is deprecated` fires on every batch. Documented in REWRITE_DESIGN.md Postscript H5 — won't fix unless PyTorch deprecates the combination. Filter with `grep -v 'deprecated'` if it's noisy.

12. **PID returned by `Start-Process` (PowerShell on laptop) is the bash wrapper, not the python**: SSH's `& echo $!` prints the inner-shell PID. To find the actual python: `ps -ef | grep train_bc` on the pod. PIDs don't matter much — kill the parent and python dies via SIGHUP unless `disown` was used.

13. **Mid-step ckpt files are 240 MB each** (model + optimizer + scheduler full state). At `--save-every 1000` and ~30 min/epoch, that's ~5-10 saves per epoch. 50 epochs would write ~50-100 ckpts = 12-24 GB. Manageable on 150 GB pod disk; clean up old mid_steps periodically if disk fills.

14. **`disown` is needed after `nohup ... &`** on RunPod because the SSH session's bash is tied to the foreground process group otherwise. Without it, closing the SSH session can SIGHUP the python. With it, training survives session termination cleanly.

### Cost quirks

15. **$1.50/hr A100 SXM 80GB beats $1.69 A100 PCIe 40GB** on every axis: more VRAM, higher memory bandwidth, lower price. Always pick SXM if available.

16. **R2 egress is free** (Cloudflare's main differentiator vs AWS S3). We can sync from R2 to RunPod (different networks) without egress fees. AWS S3 would charge $0.02/GB for cross-region.

17. **Pod billing continues during data sync**: A 24 hr local→cloud sync at $1.50/hr would cost $36 in idle GPU time. **Always upload to R2 first**, then sync R2→pod (which is fast). Never rsync laptop→running-pod for big data.

## 10. Optimization knobs (validated Session 48)

Things we proved work / don't work on this exact hardware:

| Knob | Default | Effect | Our finding |
|------|---------|--------|-------------|
| `--batch-size 48` | yes | per-step throughput | Saturates A100 at ~4 ms/turn fp16. Going higher wastes memory. |
| `--workers 8` | yes | DataLoader parallelism | Sufficient on RunPod. Going to 16 doesn't help (compute-bound). |
| `--compile` | yes | torch.compile spatial + temporal | ~10-25% speedup. ~30s warmup. Worth it for runs > 1 hr. |
| `--fp16` | yes | AMP mixed precision | ~50% memory savings. No quality regression. Required for B=48 at decent throughput. |
| `--gradient-checkpoint` | NO on cloud | activation memory savings | Only needed on 6 GB local GPU. **Skip on 80 GB A100** — adds ~30% slowdown for no gain. |
| `--sched constant` | yes | constant LR | Matches Metamon. Cosine has eta_min=lr default which is effectively constant anyway. |
| `--early-stop-patience 2` | yes | exit on plateau | Metamon convention. Saves compute. |
| `--lr 1e-4` | yes | learning rate | Metamon-validated for B=48. Don't sqrt-scale to higher batches; BC is not LR-sensitive. |
| TF32 (`set_float32_matmul_precision("high")`) | enabled in train_bc.py | 2-4× fp32 matmul on Ampere | Yes — train_bc.py has this baked in at module load. |
| cuDNN benchmark (`torch.backends.cudnn.benchmark=True`) | enabled in train_bc.py | autotune kernels for stable shapes | Yes — train_bc.py has this baked in. |
| `--bf16` | not implemented | could replace fp16 | Maybe cleaner numerical stability. Add if fp16 ever shows underflow issues. |
| larger LR + warmup | not used | maybe faster convergence | Not needed for V1; Metamon's 1e-4 constant works. Bump only if val_loss stalls early. |

### Throughput we've measured (Session 48)

- A100 SXM 80GB + B=48 + fp16 + compile: **~4 ms/turn**
- Per epoch: **~5.5 hr** for 5M turns at B=48 (3750 batches/epoch)
- Per-batch wall-clock: ~5-6 s steady state; ~30s warmup on first batch

**5-epoch BC at this rate: ~28 hr ≈ $42.** With early-stopping firing at epoch 3-4, more like **20-25 hr ≈ $30-38**.

Per-epoch loss/acc trajectory we've validated:
- Epoch 0: train_acc 0.55, val_acc 0.62 (smart_avg ~60% if eval'd here)
- Predicted epoch 1: val_acc ~0.63-0.65 (smart_avg ~63-66%)
- Predicted epoch 2-3: peak, val_acc ~0.65-0.68 (smart_avg ~65-70%)

## 13. Where things can go wrong + diagnostic commands

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| SSH "Permission denied (publickey)" | Pubkey not on pod | Web terminal → append to `/root/.ssh/authorized_keys` |
| `aws s3 ls ...` AccessDenied | R2 token scoped to bucket | Use `aws s3 ls s3://team-builder-data/ ...` (specify bucket) |
| Tests fail with `FileNotFoundError: data/vocab/species.json` | scp'd vocab missing | scp from laptop (section 3d) |
| Tests fail with `FileNotFoundError: data/lookup/move_flags_v1.pt` | same | same |
| `cloud_smoke.sh` says FAIL but log shows clean training | Smoke's 10-min timeout | Ignore exit code; check log content |
| Training hangs at first batch | torch.compile warmup | Wait 30-60s; if longer, kill and retry without `--compile` |
| GPU util 0% mid-training | DataLoader stall | Check `ps -ef \| grep python`; if workers crashed, drop `--workers` to 4 or 0 |
| OOM mid-training | shouldn't happen on 80 GB; if it does | Drop B to 32, or add `--gradient-checkpoint` |
| `RuntimeError: CUDA driver` | Pod hardware issue | Terminate + redeploy on different host |
| Runs but no checkpoint files | `--save-every` flag too high or path wrong | Check `ls .../v10_cloud_gen9/`; expect `mid_step*.pt` every ~5-10 min |
| Test loads wrong arch | `is_transformer_checkpoint` matched legacy | Verify it uses `tokenizer./switch_encoder./action_head.` only (Session 48 fix) |
