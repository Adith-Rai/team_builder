## Architecture context (verified)

| Aspect | Old arch (PokeTransformer, model.py) | New arch (TransformerBattlePolicy, model_transformer.py) |
|---|---|---|
| Tokenization | Aggregated: 12 pokemon tokens + 1 field + 1 transition = 14 tokens. PokemonNet pools all attributes per mon into one token. | Per-attribute: ~220 tokens. Each pokemon's species, item, ability, stats, moves all separate tokens. Plus field + transition + scratch. |
| Param count | 12-15M | 19,994,924 |
| d_model | 384 (spatial) | 256 |
| d_temporal | varies | 512 |
| Spatial layers | 4-6 | 6 |
| Temporal layers | 2 | 4 |
| Positional info | Implicit (pooled mons) | Explicit `type_id` embedding per token slot |

Per-attribute tokenization is fundamentally different from aggregated. **Most things tuned for the old arch may not transfer 1:1 to the new arch.** This includes hyperparameters, data pipeline assumptions, PPO machinery, and possibly even eval setups.

## Hypotheses ranked by current belief

### H1: Hyperparameter sensitivity differs on new arch — HIGH likelihood
PPO hyperparams are notoriously arch-sensitive. lr=3e-5 + clip=0.2 + ppo_epochs=5 was validated on the 12-15M old arch. The 20M per-attribute new arch may have different optimal hyperparams. Specifically:
- KL spike to 0.045+ every iter for 79 iters compounds into massive cumulative policy drift
- ppo_epochs=5 may be too aggressive when each forward pass is heavier (220 vs 14 tokens × deeper attention)
- ent_coef=0.02 may be wrong for a model with different baseline entropy

**Test:** ablate lr (1e-5, 5e-6), ppo_epochs (2, 3), ent_coef (0.01, 0.005). Run 1-iter smoke each, look for SE-attack rate after a few iters.

### H2: Feature pipeline divergence between BC training and PPO training — HIGH if real
- BC trained on memmap'd `make_features` output (generated by `replay_to_memmap.py`)
- PPO trains on live `make_features(battle)` output during V9RLPlayer collection
- They SHOULD be numerically identical batches but I haven't verified

**Specific suspicions:**
- `our_pokemon_move_ids` source: BC has them deterministic from replays; PPO has them evolve as battle reveals (initially missing, fills in over turns)
- Opponent stats: BC has recorded final stats; PPO live has poke-env's level-100 estimates
- `move_flag_lookup` versioning: BC was trained against one version, PPO loads possibly another
- Unicode/normalization edge cases that differ between memmap pre-processing and live extraction

**Test:** Load a single battle replay state, run through both pipelines, diff the resulting batch dicts at tensor level. **Cheapest definitive test we can run.** ~30 min.

### H3: InferenceBatcher dispatch produces slightly different logits than BattleAgentTransformer — UNTESTED
- Eval (BattleAgentTransformer) calls `model.forward(batch, history=...)` directly — full path
- PPO collection (V9RLPlayer + InferenceBatcher) calls staged `forward_spatial → action_encoder_from_spatial → action_head.mlp` — refactored via `arch_compat.py` in Session 50

I refactored this in Session 50 and confirmed via smokes that it produces a real-looking iter 0 line. But I never directly compared logits from the two paths on the same input.

**Test:** Same battle state, route through both paths, diff the action_logits. If they diverge meaningfully, my arch_compat refactor produced a subtle bug. **This is the second-cheapest test to run.** ~30 min.

### H4: Per-attribute tokenization has more catastrophic-forgetting surface area — MEDIUM
- Type-effectiveness "knowledge" lives across many tokens (species, ability, type-derived features)
- Easier for noisy gradient updates to break it across that wider surface
- Old arch's pooled-pokemon token had less surface area — knowledge was more concentrated and more robust

**Test:** Logit/probability dump per snapshot. If sp79 has near-zero probability on certain action classes (status moves, recovery), mode collapse is happening. ~20 min.

### H5: Procedural teams creating degenerate self-play signal on new arch specifically — LOW
- Procedural teams worked fine on old arch for 1000s of iters
- BUT the new arch's per-attribute tokenization may be more sensitive to "team incoherence"
- Procedurals lack the structured cores that competitive teams have, and the new arch may be picking up on the lack-of-pattern in unhelpful ways

**Test:** PPO restart with mixed procedural+competitive teams. But this is a real-run test (~hours), defer until cheap tests are exhausted.

### H6: The old severe-dip-then-rise pattern was lr-driven — VERIFIED with caveats
The user recalled an early run that dipped deep below BC then rose past it, eventually reaching 50-66%.

**FINDING (Session 50 STATUS.md:541-545 archaeology):** That historical run used:
- **lr = 1e-4** (3.3x higher than our 3e-5)
- **ent_coef = 0.04** (2x higher than our 0.02)
- **lam = 0.75** (much shorter advantage horizon than our 0.95)
- gamma = 0.9999, ppo_epochs = 5, clip_eps = 0.2 (same as ours)

**Trajectory at those hyperparams:**
- BC_base: Elo 809 (start)
- snapshot_0014: Elo 795 (-14 from BC, early dip)
- snapshot_0194: Elo 729 (**-80 from BC, deep dip**)
- snapshot_0284: Elo 951 (**+222 from snapshot_0194, type-effectiveness features added in Session 30**)
- snapshot_0589: Elo 1019 (+68 more)
- snapshot_1784: Elo 1029 (final, plateau era)

**KEY INSIGHT:** The "rise" wasn't self-recovery from training — it was **feature engineering**. Session 30 added type-effectiveness features → +222 Elo jump. The dip pattern at lr=1e-4 didn't auto-correct; it stayed dipped until features changed.

**WHY OUR LR=3E-5 IS NOT THE OLD WORKING CONFIG:**
- Per `next-prompt.txt`: lr=3e-5 was set because "1e-4 has diverged in 3 iters in every prior attempt (S39, S43)."
- BUT S39/S43 were on the LEGACY arch (PokeTransformer, 12-15M). The new arch (TransformerBattlePolicy, 20M, per-attribute) has never been tested at any lr.
- Phase 1 inherited lr=3e-5 as "the safe choice" but this is a triple-novel configuration: new arch + new BC v10 + lr from legacy validation.

**Updated prediction:** at lr=3e-5 on the new arch, gradient updates may be too small to escape the local minimum the model drifts into. The old run had high-amplitude noise that allowed escape; we have low-amplitude consistent drift in a bad direction. **Hyperparameter sensitivity is the most likely root cause** (now H1, was tied with H2).

## What we should NOT do

- ❌ Run more iters of the same Phase 1 setup hoping it dips-then-rises. The trajectory is monotonically down, not oscillatory. More iters = worse model.
- ❌ Drop procedural teams. They've been validated for 1000s of iters on old arch. The procedure is sound; the arch interaction is the variable.
- ❌ Add mcts-fast or other Phase 2 changes before understanding the Phase 1 failure. We'd be conflating fix-attempts with diagnosis.
- ❌ Re-run old-arch as a primary diagnostic. Time-expensive; the registry data already shows the old arch works under same procedure. If we need definitive proof later, fine, but not as Tier 1.

## Test results so far (Session 50 continuation)

### ✓ Pri 1 (Feature pipeline equality) — RULED OUT H2

`feature_equality_test.py` ran on a real h2h replay across 12 turns. Both `_feat_to_flat_record` (BC pipeline) and `build_turn_batch` (PPO pipeline) produce **bit-identical** batch dicts when given the same `feat`. **Max diff = 0** across all fields, all turns.

Notable structural difference reconciled by `_fix_ids` (model_transformer.py:1531): BC stores `our_pokemon_ids` as (6, 7) (species/item/ability/move0-3 all together), PPO splits as (6, 3) + separate (6, 4) `our_pokemon_move_ids`. Model accepts both, concatenates to (6, 7) internally. Verified: when slicing the 7-wide form to compare against the 3+4 split, all values match exactly.

**Conclusion:** No feature pipeline bug. Model sees the same inputs in BC and PPO contexts.

### ✓ Pri 1.5 (InferenceBatcher staged path vs `model.forward()`) — RULED OUT H3

`inference_batcher_equality_test.py` loaded `snapshot_0019.pt` and ran 12 turns through both:
- `model.forward(batch, history)` — eval-time direct path
- `forward_spatial → call_action_encoder → call_policy_logits → call_value_logits` — training-time staged path through arch_compat

**Max diff in action_logits, values, summary, v_logits = 0** across all turns. The Session 50 arch_compat refactor produces **bit-identical** output to direct forward. No refactor bug.

**Conclusion:** PPO trained on the same logits eval reads. Refactor isn't the bug.

### Implication

Both code-correctness hypotheses (H2, H3) are ruled out. The bug must be in **how the gradient evolves the model**, not in inputs/outputs. That points to:
- **H1 (hyperparameter sensitivity for new arch)** — strongest remaining candidate
- **H4 (per-attribute tokenization → catastrophic forgetting on noisy gradients)** — possible, untested
- **H6 (lr=3e-5 too small for new arch's gradient regime)** — strongest specific instance of H1
