THE GOAL — read this every time you make a decision
═══════════════════════════════════════════════════════════════════════════════

We are building a Pokemon battle AI that competes at an elite level — among
the strongest human players, the strongest published bots (Metamon's Kakuna,
SmallRL, Minikazam), and the strongest other engines.

**Ultimate scope (load-bearing):** elite play across **every legal Pokemon
format** — singles, doubles, triples, 4v4, 6v6, all tiers (OU, Ubers, AG,
UU, RU, etc.), random battles (which have varied levels), in-game/cartridge
battles. All gens (4-9+, expanding as new gens release). Plus team
building, not just play.

**Deployment scope:** must scale **massively on paid cloud** for throughput-
heavy training AND remain **runnable locally** on a 6 GB consumer GPU for
development + smoke tests. Same code, both environments. Not parallel
codebases. See `memory/project_vision_scope.md` for full constraints.

V1 baseline is singles gen 9 — but every refactor decision must NOT
foreclose paths to the broader scope. If a shortcut hardcodes gen9ou or
6v6, surface the tradeoff explicitly before shipping.

Elite Pokemon play is about long-horizon planning, opponent modeling,
risk-adjusted decision making, and squeezing every fractional advantage
from type/STAB/effectiveness/speed/items/abilities/positioning.

BEST PRACTICES — re-read before each non-trivial decision
═══════════════════════════════════════════════════════════════════════════════

**Lessons paid for in prior sessions (don't repeat):**

1. **Read docs + code BEFORE estimating scope.** Session 49 burned an
   evening because I called PPO Phase 1 "30 min smoke + launch" based on
   grep matches; reality was a 6.5 hr refactor uncovered only by smoke.
   The codebase is bigger than your grep cache. Trust line-cited audits,
   not first-impression reads.

2. **Verify before recommending from memory.** A memory record naming a
   function/flag is a claim about a past moment, not current state.
   `is_transformer_checkpoint()` originally false-matched legacy because
   both arches share `spatial.*`/`temporal.*` prefixes — the discriminator
   is `tokenizer.`/`switch_encoder.`/`action_head.` only. If memory says
   X exists, grep for it before promising it works.

3. **Smoke ALWAYS before launching long runs.** Session 49's smoke caught
   a bug that would have wasted 40+ hours of self-play if launched blind.
   The smoke is non-optional even when "it should just work." Also smoke
   on BOTH arches when changes touch the dispatch — legacy regression is
   the silent killer.

4. **Concurrency for self-play is 200-250, not 6.** The "6" in old docs
   was for runs WITH external opponents (FP/MM subprocess limits). Pure
   self-play has been 200-250 historically per user. Check user before
   trusting doc-default conc values.

5. **Cloud: Network Volume, not Container Disk.** Pause/resume on the
   same host preserves Container Disk; stop/start can land on a different
   host and your `/workspace` is gone (104 GB memmap and all). Documented
   in `CLOUD_RUNBOOK.md §9` after Session 49 lost a partial run this way.

6. **Don't rush refactors under time pressure.** "Just ship it tonight"
   is the pattern that produces next session's surprise gaps. Better to
   stop at a clean breakpoint and resume fresh.

7. **Backward compat is a hard constraint, not a nice-to-have.** Legacy
   PPO ckpts (s39 peaks, all sp_NNNN snapshots, iter_0119) must remain
   loadable as PFSP opponents in every future run. The legacy
   `BattleAgent` / `SelfPlayOpponent` classes must keep working unchanged.
   Adding new dispatch is fine; replacing legacy paths is not.

**Scope-respecting habits:**

- Before any refactor, ask: "does this work for doubles? for randoms with
  levels? for gen 4? for 11M-param vs 100M-param models?" If not, surface
  the tradeoff before shipping.
- Anything format-specific must come from `cfg.format_config` (the
  `FormatConfig` infra exists from Session 34's refactor). Don't hardcode
  `team_size=6`, `n_actions=9`, `n_moves=4`.
- Anything gen-specific must be a per-gen lookup (move banks, ability
  lookups, item tokenizers) even if only gen9 is currently populated.

**Tone / cadence:**

- Be terse. The user values direct technical responses over hedging.
- State results and decisions directly; user-facing text is for relevant
  updates, not internal deliberation.
- When uncertain, say so explicitly — don't fake confidence.
- Honest takes are valued, especially when they push back on user
  assumptions or your own prior recommendation.
