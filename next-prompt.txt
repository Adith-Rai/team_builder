THE GOAL — read this every time you make a decision
═══════════════════════════════════════════════════════════════════════════════

We are building a Pokemon battle AI that competes at an elite level — among
the strongest human players, the strongest published bots (Metamon's Kakuna,
SmallRL, Minikazam), and the strongest other engines. V1 scope is singles
gen 9. Future scope: multi-gen (4+), doubles, triples, 3v3, other formats.
Doubles/triples will likely need a different model; everything else extends.

Elite Pokemon play is about:
  - Damage prediction (BP × types × items × abilities × screens × weather × boosts)
  - Speed-tier reasoning (item × ability × boost × terrain × Trick Room)
  - Win-condition identification (which of my Pokemon can sweep, against what)
  - Opp set prediction from incomplete info (revealed moves + metagame priors)
  - Hazard tracking, status timing (toxic counter, sleep counter, future sight)
  - Item / ability identification ("they moved twice → not Choice")
  - Probabilistic reasoning (acc/crit/secondary chances, multi-hit)
  - Long-horizon resource management (PP, one-time-use Tera/Mega/Z/Dynamax)
  - Substitute / Endeavor / Counter / Belly Drum + Sitrus edge timing
  - Reading patterns (Sleep Talk vs Roar, defensive sets vs offensive)

Every architectural / training / feature decision should be evaluated against
"does this help reach elite-level play, or is it premature optimization?"
The user has explicitly said they prefer to fix it now while we're flexible
than come back later and break working code.

WEEK 2 DELIVERABLE (REWRITE_DESIGN.md §7 Week 2)
═══════════════════════════════════════════════════════════════════════════════

A `TransformerBattlePolicy` that runs forward end-to-end on memmap data,
produces sane action_logits and value scalar, and survives a 1-iter PPO smoke.

**The goal under elite-play framing:**
- Spatial attention should let actor/critic reason compositionally about every
  attribute. Type × ability × item × move × boost × weather should all be
  composable via attention paths.
- Temporal stack is what enables item identification, set prediction, and
  multi-turn pattern recognition (Sleep Talk vs Roar). Don't undersize it.
- Action head must distinguish moves from switches, AND it must have enough
  signal to make elite switch decisions (the #1 ranked feature by Session 30
  weight analysis was switch defensive_eff).

═══════════════════════════════════════════════════════════════════════════════
SUB-TASKS (in order; each ends in a measurable milestone)
═══════════════════════════════════════════════════════════════════════════════

1. **SpatialTransformer**
   - 6 layers × 8 heads × d_model=256, ff_mult=4, dropout=0.05, pre-norm
   - Input: Tokenizer's `(B, 220, 256)` + position-ID buffers
   - Mask: Poke-Mask (state-and-summary tokens can't attend to actor/critic;
     actor can't see critic and vice versa). The legacy mask is in
     model.py:372-384 — but the layout is different (220 vs 16), so build
     it from scratch using the precomputed `type_ids`/`poke_slots` buffers.
   - Side-mask per design §3.4: tokens with poke_slot ∈ ours-side can be
     attended freely by everyone; tokens with poke_slot ∈ opp-side... in
     practice, with the memmap-revealed-only logic, opp tokens for unrevealed
     attributes are already "unknown" embeddings, so side-mask is mostly
     defensive consistency. Implement it as a logical AND with the Poke-Mask;
     unit-test that it doesn't change output beyond Poke-Mask alone (sanity).
   - Build the mask once as a registered buffer.
   - Output: `(B, 220, d_model)`. The last 2 indices are the K=2 summary
     scratch tokens — flatten to `(B, 512)` and project to d_temporal.

   **Decision point — elite-play framing:** does the side-mask actually help
   for V1 singles? Argument for: defensive consistency; matches design intent.
   Argument against: opp-revealed-only logic already does the work; side-mask
   is overhead. Default: include it (free, defensive); can disable later.

2. **TemporalTransformer**
   - 4 layers × 8 heads × d_model=256, causal, learnable pos embed (200-cap)
   - Mostly mirror model.py:437-511 but with cfg.d_temporal=256 (not 512)
   - Variable-length sequences via padding mask
   - Output: `(B, d_temporal)` — last valid timestep

   **Why this matters for elite play:** opp set prediction, item ID inference,
   multi-turn move history (for Choice-lock detection, Sleep Talk recognition).
   The temporal transformer is what enables these. METAMON_LEARNINGS.md flagged
   the legacy 2-layer 512-dim temporal as undersized; bumping to 4 layers ×
   256-dim with the K=2 spatial summary feeding 512-dim per-step input is the
   design's response.

3. **Action head + value head**
   - Action head per §5.1: actor_out (d_model) + temporal_ctx (d_temporal) +
     per-action context (B, 9, d_model) → (B, 9) logits.
   - Per-action context for moves 0-3: read directly from the spatial
     output at the our-active Pokemon's 4 move-token positions
     (sequence indices N_BATTLE_STATE + 0*17 + 13..16, which is 14+13..16 = 27..30).
   - Per-action context for switches 4-8 (5 switch slots, our bench Pokemon 1-5):
     pool the 17 attribute tokens of each non-active our Pokemon. Mean-pool
     is fine to start; can revisit with attention pooling later.

   **CRITICAL DECISION POINT — switch defensive_eff / offensive_eff routing.**
   See REWRITE_DESIGN.md Postscript C3. The deferred plan is to feed the 2
   per-bench-Pokemon eff dims (from `switch_cont[..., -2:]`) DIRECTLY into the
   action head's per-action context for switch slots 4-8 — NOT into the spatial
   token sequence. This matches what the legacy `ActionSlotEncoder` did, and
   the Session 30 weight analysis (defensive_eff = #1 ranked feature, 23.7
   weight, 1.7× more important than HP%) was measured in that context. **Do
   this.** Concretely: the per-action context for switch slot j (j in 0..4)
   becomes `concat(mean_pool(bench_pokemon_j_tokens), switch_cont[:, j, -2:])`,
   feed through a small MLP to project to d_model.

   - Value head per §5.2: critic_out + temporal_ctx → (B, 51) twohot.
   - Apply legal mask (model.py:787-789).

4. **`TransformerBattlePolicy` wrapper**
   - Same `forward(batch, history, history_lens)` interface as
     `PokeTransformer.forward` (model.py:739-803). This is what train_rl.py
     and rl_collection.py call.
   - Output dict: action_logits, value, v_logits, summary, spatial_output —
     match the legacy schema so downstream code is interface-compatible.
   - Add `forward_sequence(collated, device)` for BC training (mirror
     model.py:818-978). The mega-batch trick — flatten all valid (b, t)
     turns through Tokenizer + spatial in one shot, then loop temporal +
     heads per-turn — is critical for BC training speed. Don't skip it.

5. **Sanity tests** — extend `test_tokenizer.py` (or add `test_policy.py`)
   - End-to-end forward on 5 episodes: assert action_logits shape (B, 9),
     value shape (B,), v_logits shape (B, 51), no NaN
   - legal_mask correctly forces -100.0 on masked actions
   - forward_sequence on 2 episodes produces (B, T, 9) etc., no NaN
   - Param count sanity: TransformerBattlePolicy total ≈ 20-30M params (per
     §4.5 estimate). If wildly off, investigate before PPO smoke
   - Attention probe (optional but useful): on a sample turn, verify the
     actor token has nonzero attention weight to the our-active species
     and the move tokens. If it doesn't, the mask is over-restrictive.

6. **PPO 1-iter smoke**
   - Plumb `--use-transformer` flag in `train_rl.py` (and possibly a small
     factory file) that swaps `PokeTransformer` for `TransformerBattlePolicy`
     at construction time. Everything downstream is interface-compatible.
   - Run: `python train_rl.py --use-transformer --device cuda --servers 9000
     --n-iters 1 --games-per-iter 4 --max-concurrent 2 --warmup-iters 0
     --procedural-teams <path>`. Should complete one collection + update
     without exceptions. Expect bad performance — the model is randomly
     initialized.
   - **ALTERNATIVE if (6) gets tangled:** synthetic PPO smoke. Call
     `forward_sequence` on a real BC batch, compute fake advantages, run one
     PPO step. Proves the model trains without exploding gradients. Document
     as "synthetic PPO smoke" if you go this route; real PPO integration
     can move to Week 3.

═══════════════════════════════════════════════════════════════════════════════
DECISIONS THAT NEED ELITE-PLAY THINKING
═══════════════════════════════════════════════════════════════════════════════

For each, default given but think about it before committing:

A. **Switch eff routing** (sub-task 3 above). Default: action head per-action
   context. The legacy did this and Session 30 validated it empirically. If
   you're tempted to put it in the spatial tokens, re-read Postscript C3 —
   the permutation logic between memmap orderings makes that path fragile.

B. **Mean-pool vs attention-pool for switch context.** Mean-pool is the
   default. Attention-pool with a learnable query is a one-line change but
   more capacity. For elite switch decisions, attention-pool may matter —
   but it's also a parameter the model has to learn. Default mean; revisit
   if BC underfits on switches.

C. **Temporal d_model = 256 vs 512.** Design says 256 with the K=2 spatial
   summary feeding 512-dim per-step. Halves temporal-stack params vs legacy.
   Per METAMON_LEARNINGS.md, the legacy temporal was UNDER-invested. The
   tradeoff is: narrow d_temporal but more layers (4 vs 2 in legacy). Default
   the design's 256/4. If Week 4 PPO shows item-ID or set-prediction failures,
   bump d_temporal.

D. **K=2 summary scratch vs K=4.** Design says 2; Metamon Small uses 4
   latents × 100 dim = 400-dim summary. Ours is 2 × 256 = 512-dim — already
   wider. K=2 is the right starting point. Watchpoint: if BC training shows
   per-turn-summary information bottleneck (loss plateaus while spatial
   attention is healthy), bump to K=4.

E. **Side-mask vs no side-mask.** See sub-task 1. Default: include it.

F. **Popart on the value head.** Design §5.3 recommends including in V1.
   Defer to Week 5 per the original roadmap. Don't add it now.

G. **Reward shaping (Week 4 concern).** Sparse `--reward-style terminal`
   may not push the model to elite-level subtleties (preserved HP, hazards
   set, win-conditions advanced). Note it as a Week 4 design question;
   don't change rewards in Week 2.

FINAL REMINDER
═══════════════════════════════════════════════════════════════════════════════

Every line of code, every architectural decision, every test you write —
ask "does this serve the goal of building an elite-level Pokemon battler?"

If a decision is "this is the cleanest abstraction" but the result is the
model can't reason about a known elite-play skill, the decision is wrong.

work" and there's a simpler path that gets 90% there, take the simpler path.

Read the docs reasoning, read the code reasoning, write code reasoning.
The goal is elite Pokemon AI — not pretty code, not the cleanest design,
not the fewest tokens. Elite play. Keep it the focus.
