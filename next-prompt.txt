WEEK 2 DELIVERABLE (from REWRITE_DESIGN.md §7)
═══════════════════════════════════════════════════════════════════════════════

`TransformerBattlePolicy` runs forward end-to-end, produces sane
action_logits and value scalar on memmap data. PPO 1-iter smoke passes.

Sub-tasks (in order):

1. **SpatialTransformer in `model_transformer.py`.**
   - 6 layers × 8 heads × d_model=256, ff_mult=4, dropout=0.05, pre-norm.
   - Input: the Tokenizer output (B, 212, d_model) + the position-ID buffers.
   - Mask: Poke-Mask + side-mask per REWRITE_DESIGN.md §3.4 / §4.1. The legacy
     impl is at model.py:312-430; the layout is different (212 tokens vs 16),
     so build a NEW mask from scratch using the position-ID buffers.
     Concretely:
       - Decision tokens are TT_ACTOR (0) and TT_CRITIC (1), at sequence
         indices 0 and 1 by construction.
       - state-and-summary tokens cannot attend to actor/critic.
       - actor cannot see critic and vice versa.
       - Side mask: tokens with poke_slot ∈ {0..5} are "ours", {6..11} are
         "opp", {12..15} are battle-state/neutral. The hidden-vs-revealed
         distinction in §3.4 is moot (memmap already zeroes unrevealed opp
         info), so side mask is mostly defensive — implement it but unit-test
         that it doesn't change output beyond the Poke-Mask alone (sanity).
     Make the mask a registered buffer; build it once in __init__.
   - Output: (B, 212, d_model). The summary scratch tokens (last 2 indices)
     are the per-turn summary; flatten + project to d_temporal.

2. **TemporalTransformer.**
   - 4 layers × 8 heads × d_model=256, causal mask, learnable pos embed
     (200-position cap). Mostly mirror the legacy class at model.py:437-511
     but with cfg.d_temporal=256 (not 512). Variable-length sequences via
     padding mask.
   - Input: (B, T, d_temporal) per-turn summaries.
   - Output: (B, d_temporal) — the last valid timestep.

3. **Action head + value head (mostly preserved).**
   - Action head per §5.1: actor_out (d_model) + temporal_ctx (d_temporal)
     + per-action context (B, 9, d_model) → (B, 9) logits. Per-action
     context: 4 active moves come straight from the spatial output (their
     token positions are the our-active Pokemon's 4 move slots). 5 switch
     slots: pool the 17 tokens of each non-active our Pokemon (start with
     mean pool; revisit if BC underfits).
   - Value head per §5.2: critic_out + temporal_ctx → (B, 51) twohot.
   - Apply legal mask exactly like model.py:787-789.

4. **`TransformerBattlePolicy` wrapper.**
   - Same forward(batch, history, history_lens) interface as
     `PokeTransformer.forward` (model.py:739-803), so train_rl.py /
     rl_collection.py / battle_agent_transformer.py can swap it in.
   - Output dict keys: action_logits, value, v_logits, summary (for
     temporal buffer), spatial_output. Match the legacy schema.
   - Add forward_sequence(collated, device) for BC training (mirror
     model.py:818-978). Big win: batch all valid (b, t) turns through
     spatial+tokenizer in one go.

5. **Sanity tests.**
   - Extend `test_tokenizer.py` (or add `test_policy.py`) with:
     - End-to-end forward on 5 episodes — assert action_logits shape
       (B, 9), value shape (B,), v_logits shape (B, 51), no NaN.
     - legal_mask correctly forces -100.0 on masked actions.
     - forward_sequence on 2 episodes produces (B, T, 9) etc., no NaN.
     - Param count sanity: TransformerBattlePolicy total params should
       be ~25M ± 5M per §4.5 estimate. If wildly off, investigate before
       PPO smoke.

6. **PPO 1-iter smoke (§7 Week 2 last task).**
   - This requires plumbing in `train_rl.py` / `rl_collection.py` /
     `battle_agent.py` to be able to construct a TransformerBattlePolicy
     instead of a PokeTransformer. The MINIMAL plumbing is a flag like
     `--use-transformer` that swaps the model class at construction time
     (everything downstream is interface-compatible).
     The §6b guard rail says don't modify model.py / features.py /
     battle_agent.py. So:
       a. Add `--use-transformer` to train_rl.py CLI.
       b. Add a model factory function in train_rl.py (or a small new
          file `policy_factory.py`) that picks PokeTransformer or
          TransformerBattlePolicy based on the flag.
       c. For BC init: skip — Week 2 just needs random-init PPO smoke.
          For init-from arg: support either checkpoint format; Week 2
          uses random init.
       d. Run: `python train_rl.py --use-transformer --device cuda
          --servers 9000 --n-iters 1 --games-per-iter 4 --max-concurrent 2
          --warmup-iters 0 --procedural-teams <path>`. Should complete
          one collection + update without exceptions.
   - ALTERNATIVE if (6) gets too tangled: stub the PPO smoke with a
     simulated batch (no battle server) — call forward_sequence on a
     real BC batch, compute fake advantages, run one PPO step. This
     proves the model trains without exploding. Document as "synthetic
     PPO smoke" if you go this route. The real PPO integration can move
     to Week 3.
