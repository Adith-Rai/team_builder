WEEK 1 DELIVERABLE (from REWRITE_DESIGN.md §7)
═══════════════════════════════════════════════════════════════════════════════

`pokemon-ai-starter/pokemon-ai/src/model_transformer.py` exports `Tokenizer` that
consumes the existing memmap batch format (see dataset.py:167-285 for shape
contract) and emits a `(B, 212, d_model)` tensor + slot/type ID tensors for
positional embeddings.

Sub-tasks (in order):

1. **Create `model_transformer.py` skeleton.**
   - `TransformerConfig` dataclass mirroring `PokeTransformerConfig` (model.py:33)
     but with new defaults per REWRITE_DESIGN.md §4: d_model=256, n_spatial=6,
     n_temporal=4, n_heads=8, n_summary_tokens=2, dropout=0.05, ff_mult=4.
   - Stub `Tokenizer`, `MoveTokenizer` modules. Don't implement the bodies yet;
     just get the imports + class signatures landing.
   - Add to `model_transformer.py` (NOT model.py — leave that untouched per §6b).

2. **Implement `MoveTokenizer` per REWRITE_DESIGN.md §3.2.**
   - Inputs: move_id, bp/acc/pp/prio bank lookups, 107-dim flag vector.
   - Module: 187-dim concat → 2-layer MLP → d_model.
   - Test: instantiate, run a (B=4, n_moves=4) forward, assert output shape
     `(B, n_moves, d_model)`, no NaN.

3. **Build the `(n_moves, 107)` flag lookup table.**
   - For each move_id 0..n_moves-1, run `_project_move_flags(move)`
     (features.py:1005-1263) and store the 107-dim "continuous" output.
   - Save to disk as `data/lookup/move_flags_v1.pt` (or similar). Build script
     in `model_transformer.py` or a separate `build_move_flag_lookup.py`.
   - **Critical verification step:** sample 50 moves at random; for each, build
     a fake battle with that move on the active Pokemon, run features.py's
     active path (features.py:_encode_action_slots ~line 1346), and assert the
     107-dim output matches the lookup table to within fp32 noise. If it
     doesn't match, the lookup is wrong — the 107-dim flags may have
     battle-state-dependent components we missed. (See REWRITE_DESIGN.md §6.1
     "Option A vs Option B" — if Option A fails, fall back to B.)

4. **Implement `Tokenizer` per REWRITE_DESIGN.md §3.1 + §6.1.**
   - Consumes the memmap batch dict (the same shape `_encode_entities` consumes
     today, model.py:668-720). Outputs `(B, 212, d_model)` + slot/type IDs.
   - For each of the 17 token types per Pokemon, slice the appropriate fields
     from the existing memmap tensors. Use the dim breakdowns in the
     features.py audit (in REWRITE_DESIGN.md §3.1 / §6.1 table):
       - species_token: embedding lookup of `our_pokemon_ids[..., 0]`
       - item_token: embedding lookup of `our_pokemon_ids[..., 1]`
       - ability_token: embedding lookup of `our_pokemon_ids[..., 2]`
       - type_token: small MLP from 2× type embeds (recover via top-2 of
         `our_pokemon_cont[..., 0:19]`)
       - status_token: small MLP from sliced flags (status + volatiles +
         paradox + tera; see features.py audit dim breakdown)
       - hp_pct_token: NumericalBank(101) of `our_pokemon_banks[..., 0]`
       - boosts_token: small MLP from `our_pokemon_cont[..., 26:117]` 7×13 one-hot
       - 6× stat_token: NumericalBank(256) of `our_pokemon_banks[..., 4:10]`
       - 4× move_token: MoveTokenizer with move_id from
         `our_pokemon_ids[..., 3+i]`, bp/acc/pp/prio from lookup table indexed
         by move_id, 107-dim flags from the lookup table.
   - Plus 6 battle-state tokens: actor (learnable), critic (learnable), field
     (MLP from 52-dim cont + 4 banks), transition (MLP from 51-dim cont + 2
     action embeds), 2× summary scratch (learnable).
   - Plus 2 active threat tokens (REWRITE_DESIGN.md §3.2 last paragraph; bumps
     total to 212).
   - Add type_id and pokemon_slot_id and move_slot_id positional embeddings
     (§3.3). Sum into token embeddings before returning.

5. **Write unit tests in `test_tokenizer.py`.**
   - Load 5 sample turns from `data/datasets/human_v8_100k/` via
     `MemmapDataset` (dataset.py:22-164).
   - For each, assert:
     - Output shape: `(B, 212, d_model)` exactly
     - No NaN, no inf
     - type_id embeddings: pull a few specific tokens (e.g., the species_token
       for our Pokemon 3) and verify it has the type_id embedding for type 0
       added (sanity check the sum)
     - Move tokens for opp Pokemon use move_id=0 ("unknown") for unrevealed
       moves (check via the memmap's `opp_pokemon_ids[..., 3:7]` having zeros)
   - Run with: `cd pokemon-ai-starter/pokemon-ai/src && python -m pytest
     test_tokenizer.py -v`. Or just plain python if pytest isn't set up.

6. **Forward-pass benchmark.**
   - On RTX 3060 Laptop, time `Tokenizer + dummy_spatial(212, d_model=256)` on
     a batch of 32 turns. Run 10 iterations after warmup, report median.
   - Target: <50ms median. If >100ms, profile and optimize. Common culprits
     are torch.cat in a loop (use torch.stack), or the (n_moves, 107) lookup
     building inside forward (should be cached at module init).
