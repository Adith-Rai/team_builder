**STEP 2a: Worker proc respawn** (~1 day)
If a `_cis_worker_main` process dies mid-iter, the iter is lost

**STEP 2b: CIS subprocess respawn** (~half day, easy to underestimate)
If the CIS subprocess itself dies (CUDA OOM, segfault, signal kill,
unhandled exception escapes service loop), all workers' recv_loops
hit EOF, all in-flight requests fail, current iter dies, NO RECOVERY
mechanism exists. For a 5-7 day Phase 2 run, this WILL happen.

When CIS dies, parent's `_ctrl_handle.reload()` also fails. Workers'
`CISInferenceBatcher.submit` raises. Workers report errors to
orchestrator but there's no path to re-spawn CIS + reload all slots
+ retry the iter.

Implementation: at CISServer level, watchdog the subprocess. On death:
1. Workers must pause (signal them via `worker_ctrl_pipes`)
2. Re-spawn CIS subprocess (re-create pipe pairs, re-spawn `_cis_main_multi`)
3. Re-load all K+1 model slots from `current_slot_paths`
4. Re-build CISClientHandle for each worker (new pipe pairs)
5. Workers resume with new handle references
6. Retry the iter (don't lose collect progress if possible — checkpoint
   trajectories on disk per-worker, similar to mp_disk_collect)

Files:
- `mp_centralized_collect.py:CISServer` — watchdog method polling
  `self._proc.is_alive()`, respawn logic
- `mp_centralized_collect.py:_cis_wait_results_and_aggregate` — detect
  CIS death (all workers reporting BrokenPipeError simultaneously),
  trigger CISServer respawn, re-issue collect_iter to workers
- New CISClientHandle method to swap pipe references (or re-construct)

Validation: cookbook §5 Test 6b — kill -9 the CIS subprocess mid-iter,
verify CISServer respawns it + all slots reload + workers continue +
iter completes. Run against the same prod-scale config.

**STEP 3: Tier 3 transition-level minibatching** (~1.5-2 weeks dev + 20-iter cloud A/B validation)
Rationale: this is THE remaining big lever. PPO update phase is currently
bound by per-episode batching (~48 episodes/forward). Published practice
(ps-ppo, Metamon) uses 256-2048 transitions/gradient — we're 48× below.
Estimated: **4-10× update speedup** if quality holds. For 5M trajectories,
this is the difference between ~70 days and ~22-30 days post-CIS.

**Reference: Metamon's source.** `metamon_ref/metamon/rl/metamon_to_amago.py`
shows the right batch shape for transformer-temporal-with-on-policy-PPO:
- `MetamonAMAGODataset._process_data`: forms RLData with variable-length
  episodes, shape `(obs={K:(L+1,...)}, actions=(L,A), rews=(L,1), dones=(L,1))`
- `MetamonAMAGOExperiment.edit_actor_mask` / `edit_critic_mask`: pad_mask
  + missing_action_mask combine to mask loss terms per `(B, L, G, _)` batch
- `MetamonTstepEncoder` (line 477) and AMAGO's traj_encoder confirm our
  spatial + temporal split maps directly to their tstep + traj encoder

**NOT a reference: VGC-Bench** (cameronangliss/vgc-bench). Same library
stack (poke-env + PettingZoo), same LR (1e-5 base, annealing), same
self-play / FP / DO methods, same arch family (transformer encoder).
But: STATELESS per-turn — their `policy.py:AttentionExtractor.forward()`
reshapes obs to `(B, 12, -1)` with NO temporal dim. Their batch_size=512
is "free" because each transition is self-contained. Their multi-team
failure (79% W/L on 3 teams → 40% on 30 teams) is likely the stateless-
arch ceiling. Don't confuse "their LR matches ours" with "we should
copy their batching strategy" — they sidestepped the temporal-stack
batching problem by not having a temporal stack. Metamon is the right
reference because they kept temporal AND scaled batch.

**Implementation approach**: borrow the batch SHAPE from Metamon (sequence
of variable-length episodes, padded to L_max in batch, masked loss).
Don't borrow the algorithm — they use V-trace (off-policy with experience
replay), we stay strict on-policy PPO. Just bigger batches per forward.

**The work**:
- Reshape PPO update path from per-turn `(B, ...)` to per-episode `(B, L, ...)`
- Causal masking in temporal attention (no future-info leak)
- PPO loss per-timestep with pad_mask zeroing padding positions
- Trajectory storage refactor: per-turn dicts → `(B, L, ...)` tensors per episode

§I. LOAD-BEARING RULES + KNOWN GOTCHAS
═══════════════════════════════════════════════════════════════════════════════

**Cookbook §3j: POKE_LOOP threading**. `PSClient._listening_coroutine` is a
`concurrent.futures.Future` running in a SEPARATE thread (poke-env's
POKE_LOOP), not an asyncio.Task in your loop. `cancel()` is fire-and-forget
across threads. Sequential `_cancel_listener + del player + del opponent`
patterns MUST yield wall-clock via `await asyncio.sleep(1.5)` afterward,
or POKE_LOOP backs up and the worker hangs at 99% CPU. Rule applies any
time you create poke-env Players in a sequential await pattern.

**Cookbook §3k: Heartbeat starvation at iter boundary**. Workers'
synchronous `torch.load` blocks asyncio; async heartbeat coroutine can't
fire; main respawns "stale" workers. Mitigations live: 600s timeout,
0.25s stagger, liveness-thread heartbeats. Rule: any blocking syscall in
asyncio context needs a separate-OS-thread liveness signal.

**`_orig_mod` prefix on compile-wrapped state_dicts**. `torch.compile(module)`
adds `_orig_mod.` to that module's state_dict keys. `load_checkpoint`
strips it (ppo.py:397); `_resume_from_checkpoint` strips it (after Session
51 fix in `251cd14a`). When writing new code that loads a state_dict,
ALWAYS strip the prefix. Use `strict=True` so mismatches surface loudly,
not silently leave half-loaded state.

**torch.compile must happen AFTER resume**. Because compile changes the
state_dict key space, compile() must be applied AFTER `load_state_dict`.
train_rl.py is structured this way (Session 51 `251cd14a`). Don't move it.

**spawn context, not forkserver**. mp_disk_collect uses spawn explicitly
to dodge the CPython 3.11 SemLock race at N≥4 forkserver workers. Don't
switch back.

**`--mp` requires transformer arch**. Legacy BattleAgent ckpts (sp_NNNN)
work as PFSP opps via `make_self_play_opponent` factory dispatch but
cannot be `--init-from` for `--mp`.

**vm.max_map_count read-only on RunPod**. Bypass with mp.Pipe (FD-only,
no SemLock). All current code already does this.

**TF32 + cudnn.benchmark already enabled** at module top of train_rl.py
(matches train_bc.py). Don't re-enable elsewhere.

**Adaptive-entropy is ON by default in production**: low=0.65, high=0.95.
Currently entropy runs ~1.13-1.17, ABOVE the band — adaptive logic is
reducing ent_coef in response. This is normal; don't try to "fix" it.

**Architectural change from S52 memo** (CRITICAL — read carefully):

**CIS thread-safety issues surfaced + fixed (THREE iterations)**:
- **Fix 1 (Phase 4.3, S53)**: Test 4's first end-to-end run hit
  `_pickle.UnpicklingError: invalid load key` + Broken pipe. Root cause:
  `mp.Pipe.send` is not byte-atomic across threads; concurrent submits
  from one worker (player + opp at minimum) corrupted the pickle stream.
  Fix: per-handle `threading.Lock` wrapping each (send, poll, recv) cycle.
- **Fix 2 (Phase 4.4, S53)**: worker's recv-loop thread races with
  PARENT's `cis_handles[wid].reload()` reads at INIT time. Both processes
  have read access to the same resp pipe. If worker's recv_thread starts
  at __init__, it eats parent's reload responses → "invalid load key"
  on parent. Fix: `_cis_worker_main` constructs handle with
  `start_recv=False`, calls `cis_handle.start_recv_loop()` only on first
  `collect_iter` cmd (by which time parent's reload window has closed).
- **Fix 3 (Phase 4.5, S53)**: SAME race re-emerges at ITER 1+ boundaries
  — once worker's recv_loop is running (started at iter 0's first
  collect_iter), parent's reload at iter 1 start hits the same shared
  pipe and worker steals the response. Symptom: "untagged response
  dropped" warning in worker + reload timeout in parent + corrupted
  slot state. Fix: dedicated CTRL pipe between parent and CIS subprocess.
  CISServer now creates a ctrl pipe pair at spawn(); CIS service loop
  reads from worker pipes AND ctrl pipe via mp_wait. Reload calls go
  via `cis_server.reload_weights()` which routes via `_ctrl_handle`
  (sync mode, parent-only). Worker procs DO NOT have read access to
  ctrl pipe → race-free. **Use `cis_server.reload_weights()`, NOT
  `handles_main_view[0].reload()`** — see B13.

§K. DEAD PATHS (things tried + abandoned — DO NOT retry)
═══════════════════════════════════════════════════════════════════════════════

These have been investigated and rejected. Don't waste cycles re-walking:

- **`--opponent-device cpu`**: tried Session 50, broke at production scale
  (CPU 20M-param forward × conc=200 exceeds WS keepalive timeout).
  Cookbook §3h. Stay on `--opponent-device cuda`.
- **forkserver mp context**: SemLock race at N≥4 workers (CPython 3.11
  bug). Use spawn instead.
- **Torch tensor IPC across mp.Pipe**: mmap explosion at production scale.
  Numpy IPC only.
- **`--compile` on the OLD legacy arch**: not supported (different forward
  shape). Only the new TransformerBattlePolicy supports compile.
- **mode="reduce-overhead" for full forward**: cudagraph aliasing breaks
  on multi-module patterns + recompile churn at variable shapes. Use
  mode="default" for production, `dynamic=True`, `_dynamo.config.suppress_errors=True`.
- **Path 1 single-method compile** (`forward_spatial` only): covers ~40%
  of the path. Path 2 (per-submodule, all 5 modules) is the right answer.
  Don't revert.
- **Standalone `grad_accum > 1`**: Session 31 record suggests silent
  quality regression. Don't experiment standalone — Tier 3 (transition-
  level minibatching, §D STEP 3) SUBSUMES it via batch shape change.
- **`lr=3e-5`**: caused regression on transformer arch. lr=1e-5 won the
  Session 50 4-point ablation. Hold it.
- **Explicit per-gen feature gating in code**: investigated Session 51 D2.
  poke-env's API already returns False/0 for unavailable mechanics per
  gen. Adding code-level gates is dead weight.
- **Gigantamax / Doubles formats in V1**: out of scope. Tokenizer asserts
  n_active=1 (singles). Multi-active = future redesign per
  REWRITE_DESIGN.md.
- **Dual-slot CIS (S52 memo Option (i))**: explicitly tested, doesn't
  compose. The asyncio.gather opp concurrency in
  `_run_collect_in_worker_cis` means workers play multiple opps in
  parallel; with strict 2-slot, concurrent opp inferences race on
  slot 1's reload causing silent training corruption. S53 replaced
  with **pool-mirror slots** (1 player + K opp slots, K=pool size).
  Don't reopen this — pool-mirror is strictly better at every axis
  except GPU memory (which is trivial: ~3% of 80GB at K=16).
- **Removing CISClientHandle thread-safety**: BOTH paths in the code are
  load-bearing. Async-with-req_id-dispatch (Phase 4.4, default in
  workers) handles concurrent IPC; the per-handle Lock (Phase 4.3,
  retained in CISServer parent-side) handles serial reload calls.
  See `feedback_cis_thread_safety.md`. mp.Pipe.send is not byte-atomic;
  one path or the other is required.
- **Worker-side recv_loop starting at __init__ time**: races with
  parent's reload reads on the same pipe → "invalid load key"
  UnpicklingError. `_cis_worker_main` MUST defer `start_recv_loop()`
  until first `collect_iter` cmd is received. Don't move it earlier.
- **Routing parent reloads through worker handles** (e.g.,
  `handles_main_view[0].reload(...)` from Phase 4.4): races with worker
  recv_loops at iter 1+ boundaries. Worker eats parent's reload
  response (logged as "untagged response dropped"); parent's sync recv
  times out. Phase 4.5 added a dedicated ctrl pipe — use
  `cis_server.reload_weights()` which routes via `_ctrl_handle`
  (parent-only, no worker access).
- **VGC-Bench as a Tier 3 reference** (their LR matches ours, they use
  PPO + self-play, batch_size=512 transitions): explicitly reviewed
  and rejected as the right reference. They are STATELESS per-turn
  (no temporal stack — confirmed in their `policy.py` where
  `AttentionExtractor.forward()` reshapes to `(B, 12, -1)` with no
  temporal dim). Their batching freedom comes from architectural
  simplicity we deliberately don't share. Their multi-team failure
  mode (79% W/L on 3 teams → 40% on 30 teams) likely IS due to
  stateless arch hitting a ceiling. **Use Metamon's
  `metamon_to_amago.py` as the Tier 3 reference instead** — they have
  temporal + transformer + the right batch shape (sequence-batched
  with pad_mask).

essential. Don't average across gens — each gen has its own difficulty
calibration. See E3 in §D STEP 4.


═══════════════════════════════════════════════════════════════════════════════
§O. CARRY-OVER PRACTICES (lessons paid for in $$$ + wall-time)
═══════════════════════════════════════════════════════════════════════════════

Distilled from Sessions 49 / 50 / 50 cont. / 51. Each one cost real money
or hours when ignored. Don't relearn them.

1. **Read docs + code BEFORE estimating scope.** Sessions 49 + 50 + 51
   each burned evenings on "should be a quick fix" estimates. Trust
   line-cited audits. The compile diagnostic took ~2 hours of investigation
   before the right fix landed; the wrong fix would have been ~1 day.

2. **Verify before recommending from memory.** Memory snapshots are
   point-in-time, not current state. Grep for functions/flags before
   promising they work. The "AttributeError on new arch" claim was wrong
   in three places (cookbook + memory + next-prompt) — believing memory
   over the code wasted ~1 hour of investigation.

3. **Smoke ALWAYS before launching long runs.** The 6-test pattern in
   cookbook §5 is non-optional. Session 50 caught 5 bugs (V9RLPlayer
   kwarg, return-tuple shape, opp_records format, mp+pipeline GPU
   contention, --opp-cpu WS keepalive). Session 51 caught 3 more
   (compile-vs-resume order, _orig_mod prefix, heartbeat starvation).
   Each would have wasted 40+ hr in production.

4. **`--mp` is transformer-only.** Legacy `BattleAgent` ckpts (sp_NNNN)
   work as PFSP opponents (factory dispatches via
   `is_transformer_checkpoint`) but cannot be `--init-from` for `--mp`.

5. **Cloud: Network Volume MANDATORY for runs > 1 epoch.** Container
   Disk doesn't survive `podStop+podResume` if the pod re-provisions on
   a different host. Lost a partial BC run this way (Session 49). This
   matters for BC training (long jobs); for PPO production runs we use
   continuous training without stop/resume so it's lower risk, but new
   long-running jobs should default to Network Volume.

6. **Don't rush refactors under time pressure.** "Just ship tonight" is
   the pattern that produces next session's surprise gaps. Session 51's
   "go ahead with Path 2 properly" was the right call exactly because we
   refused to ship Path 1's incomplete coverage.

7. **Backward compat is a hard constraint** (until D1 broke it deliberately).
   Legacy ckpts must remain loadable as PFSP opponents — the
   `make_self_play_opponent` factory-dispatch invariant protects this.
   The D1 gen-id token addition deliberately breaks BC v10 → multi-gen
   model compatibility, which is OK because BC v11 retrains from scratch.

8. **Cloud-only bug discovery**: many bugs (SemLock race, GPU contention,
   heartbeat starvation, the iter-17 hang) only reproduce on Linux +
   RunPod + N≥4 workers + production scale. Schedule cloud debug time
   when context is fresh — these don't reproduce locally.

9. **POKE_LOOP threading rule** (cookbook §3j): any sequential code path
   in this codebase that creates poke-env Players AND calls
   `_cancel_listener` MUST yield wall-clock time afterward via
   `await asyncio.sleep(1.5)`. The `asyncio.gather` pattern is fine
   without this; sequential `await` is not. Caught the hard way in
   Session 50 cont. iter 10 hang.

10. **Heartbeat from a separate OS thread, not from asyncio coroutines**
    (cookbook §3k): when a worker does any blocking syscall (torch.load,
    big disk read, model copy), the asyncio loop is paused — async
    heartbeat coroutines can't fire. Liveness signaling MUST come from
    a thread that runs independent of asyncio. Session 51 iter 17 hang
    cost 7+ hours before this was fixed.
